# 天命共振 — 戰鬥系統技能公式完整分析報告

---

## 一、戰鬥流程總覽

目前的戰鬥系統是一個**回合制自動戰鬥**模型，最多進行 **10 回合**。每次 Tick 觸發戰鬥事件時，系統從當前地圖節點的怪物池中隨機選取一隻怪物，然後進入 `resolveCombat()` 函數進行結算。整個流程如下：

1. **選取怪物**：根據地圖節點的五行屬性和等級範圍，從 `shared/monsters.ts` 的靜態怪物表中篩選
2. **計算 Combo 加成**：呼叫 `calcSkillCombo()` 計算玩家裝備技能的 Combo 標籤加成和命格加成
3. **載入玩家技能**：從資料庫讀取玩家已裝備的技能（`agentSkills` + `gameSkillCatalog`）
4. **進入回合制戰鬥**：`resolveCombat()` 模擬最多 10 回合的攻防
5. **結算獎勵**：經驗值、金幣、掉落物品

## 二、傷害公式拆解

### 2.1 基礎傷害計算

玩家和怪物的基礎傷害公式如下：

| 角色 | 基礎傷害公式 |
|------|-------------|
| **玩家** | `max(1, round(attack × 五行倍率 × 魔法係數A × 種族倍率 − 怪物defense × 0.5))` |
| **怪物** | `max(1, round(monster.attack × 反向五行倍率 − 玩家defense × 0.5))` |

**關鍵差異**：玩家的基礎傷害受到**三重加乘**（五行、魔法係數A、種族），而怪物只受到**一重加乘**（反向五行倍率）。這意味著玩家在屬性對位正確時，傷害優勢非常明顯。

### 2.2 五行相剋倍率（`calcWuxingMultiplier`）

五行相剋遵循傳統的「相生相剋」體系：

| 關係 | 倍率 | 說明 |
|------|------|------|
| **相剋**（木→土、火→金、土→水、金→木、水→火） | **×1.5** | 傷害 +50% |
| **相生**（木→火、火→土、土→金、金→水、水→木） | **×1.2** | 傷害 +20% |
| **被剋**（反向相剋） | **×0.7** | 傷害 −30% |
| **無關係** | **×1.0** | 無加成 |

### 2.3 魔法係數 A（`calcSpiritCoeffA`）

這是參考魔力寶貝的精神比例系統。計算流程：

1. **施法者屬性數值** = 玩家對應五行屬性值（如火屬性角色用 `wuxingFire`）
2. **敵方抗性** = `max(5, round(怪物defense / 五行倍率))`
3. **精神比例** = `(施法者屬性 / 敵方抗性) × 100`

精神比例對應的係數 A 查表如下：

| 精神比例範圍 | 係數 A | 效果描述 |
|-------------|--------|---------|
| > 119 | **1.1** | 魔法加成極強 |
| 113 ~ 119 | **1.0** | 魔法加成強 |
| 105 ~ 112 | **0.9** | 正常偏強 |
| 97 ~ 104 | **0.7** | 正常 |
| 89 ~ 96 | **0.6** | 偏弱 |
| 79 ~ 88 | **0.4** | 弱 |
| 69 ~ 78 | **0.3** | 很弱 |
| < 69 | **0.1** | 極弱 |

**問題點**：目前 `skillElement` 固定為角色主屬性（`agentElement`），並未根據實際使用的技能屬性來動態切換。這代表即使玩家裝備了不同屬性的技能，魔法係數 A 的計算仍然只看主屬性。

### 2.4 種族剋制倍率

種族剋制表（10 組對應關係）：

| 剋制方 | 被剋方 | 倍率 |
|--------|--------|------|
| 靈獸系 | 亡魂系 | ×1.2 |
| 亡魂系 | 人型系 | ×1.2 |
| 金屬系 | 靈獸系 | ×1.2 |
| 人型系 | 金屬系 | ×1.2 |
| 植物系 | 水生系 | ×1.2 |
| 水生系 | 妖化系 | ×1.2 |
| 妖化系 | 植物系 | ×1.2 |
| 龍種系 | 金屬系 | ×1.2 |
| 蟲類系 | 植物系 | ×1.2 |
| 天命系 | 龍種系 | ×1.2 |

**玩家固定為「人型系」**，所以玩家天生剋制金屬系怪物（+20%），但被亡魂系怪物剋制（−15%）。

## 三、技能發動機率與 AI 判定

### 3.1 玩家技能選擇（智慧 AI 策略系統）

玩家的技能選擇是**全自動**的，由以下五層優先級策略決定：

| 優先級 | 條件 | 行為 | 發動機率 |
|--------|------|------|---------|
| **1（生存危機）** | HP < 30% 且有治癒技能 | **強制使用治癒技能** | 100%（條件滿足時） |
| **2（HP 偏低）** | HP 30%~50% 且有治癒技能 | 50% 機率治癒，50% 攻擊 | 50% |
| **3（五行剋制）** | 有攻擊技能 + 存在剋制怪物屬性的技能 | 80% 使用剋制技能，20% 隨機攻擊技能 | 80% |
| **4（最強攻擊）** | 有攻擊技能但無剋制技能 | 選擇 `damageMultiplier` 最高的技能 | 100%（條件滿足時） |
| **5（補血）** | 只有治癒技能且 HP < 70% | 使用治癒技能 | 100%（條件滿足時） |
| **6（普攻）** | 沒有可用技能（MP 不足或全部冷卻中） | 普通攻擊 | 100%（兜底） |

**技能可用條件**：
- MP 足夠支付 `mpCost`
- 技能不在冷卻中（所有技能使用後固定冷卻 **2 回合**）

### 3.2 怪物技能選擇

**這是目前系統最大的問題點。** 怪物的技能選擇邏輯極其簡單：

```
const monsterSkillIdx = Math.floor(Math.random() * Math.max(1, monster.skills.length));
const monsterSkillName = monster.skills[monsterSkillIdx] ?? "普通攻擊";
```

也就是說：
- 怪物技能只是**純隨機選擇一個名稱**，用來顯示在戰鬥日誌中
- **怪物技能不影響實際傷害計算**，傷害永遠是 `monsterBaseDmg × randFloat(0.8, 1.2)`
- 怪物沒有 MP 系統、沒有冷卻、沒有技能類型判定
- `gameMonsterSkillCatalog` 中定義的 `powerPercent`、`mpCost`、`cooldown`、`additionalEffect`、`aiCondition` **全部沒有被戰鬥引擎使用**

**換句話說，怪物技能目前只是「裝飾品」，不影響任何戰鬥數值。**

### 3.3 怪物數據來源的斷層

目前戰鬥引擎使用的怪物數據來自 `shared/monsters.ts` 的**靜態硬編碼陣列**，而後台 CMS 管理的怪物圖鑑存在 `gameMonsterCatalog` 資料庫表中。這兩套數據是**完全分離**的：

| 數據源 | 用途 | 技能格式 |
|--------|------|---------|
| `shared/monsters.ts`（靜態） | 戰鬥引擎實際使用 | `skills: string[]`（純名稱陣列） |
| `gameMonsterCatalog`（資料庫） | 後台 CMS 管理 | `skillId1/2/3`（指向 `gameMonsterSkillCatalog`） |

這代表：
1. 在後台用 AI 生成的怪物技能（存入 `gameMonsterSkillCatalog`），**不會影響實際戰鬥**
2. 後台修改怪物數值，**不會反映到戰鬥中**
3. 兩套數據可能產生不一致

## 四、技能傷害倍率與治癒公式

### 4.1 攻擊技能傷害

```
實際傷害 = round(基礎傷害 × 技能倍率 × randFloat(0.8, 1.2))
```

其中 `技能倍率` = `skill.damageMultiplier`（來自 `gameSkillCatalog.powerPercent`）。

之後還有閃避/格擋/暴擊判定：

| 判定 | 機率 | 效果 |
|------|------|------|
| **怪物閃避** | `min(0.2, 怪物速度 / (怪物速度 + 玩家速度 + 15))` | 傷害歸零 |
| **怪物格擋** | 10%（閃避失敗後） | 傷害 ×0.5 |
| **玩家暴擊** | 10%（閃避和格擋都失敗後） | 傷害 ×1.8 |

### 4.2 治癒技能

```
治癒量 = round(玩家maxHp × 0.2 × 技能倍率)
```

治癒回合中玩家**不攻擊怪物**，但怪物仍然會攻擊玩家。

### 4.3 怪物攻擊判定

| 判定 | 機率 | 效果 |
|------|------|------|
| **玩家閃避** | `min(0.3, 玩家速度 / (玩家速度 + 怪物速度 + 10))` | 傷害歸零 |
| **玩家格擋** | 15%（閃避失敗後） | 傷害 ×0.5 |
| **怪物暴擊** | 10%（閃避和格擋都失敗後） | 傷害 ×1.5 |

**注意**：玩家的閃避上限（30%）比怪物（20%）高，格擋率（15%）也比怪物（10%）高，但玩家暴擊倍率（×1.8）比怪物（×1.5）高。整體設計偏向玩家有利。

## 五、Combo 系統與覺醒倍率

### 5.1 Combo 標籤加成

當玩家裝備的技能中有 **2 個以上相同標籤**時，觸發 Combo 加成：

| 標籤 | 加成 | 類型 |
|------|------|------|
| fire | +15% | 傷害 |
| water | +12% | 傷害 |
| wood | +10% | 傷害 |
| metal | +18% | 傷害 |
| earth | +8% | 傷害 |
| aoe | +20% | 傷害 |
| burn | +25% | 傷害 |
| freeze | +20% | 傷害 |
| stun | +15% | 傷害 |
| heal | +15% | 治療 |
| passive | +10% | 傷害 |

多個 Combo 可以**疊加**（加法疊加，不是乘法）。

### 5.2 命格加成

當使用技能的元素 = 角色命格主屬性時，額外 **+20% 傷害**。

### 5.3 覺醒倍率

技能的覺醒階段（`awakeTier`）提供**乘法加成**：

| 覺醒階段 | 倍率 | 名稱 |
|---------|------|------|
| 0 | ×1.0 | 普通 |
| 1 | ×1.3 | 大招 |
| 2 | ×1.6 | 奧義 |
| 3 | ×2.0 | 神技 |

**重要**：Combo 和命格是加法疊加到 `damageMultiplier` 上，覺醒倍率是最後乘上去的。所以一個有 Combo + 命格 + 神技覺醒的技能，最終倍率可以達到 `(1.0 + 0.18 + 0.20) × 2.0 = 2.76`。

## 六、角色屬性計算公式

角色的基礎屬性由五行數值和等級決定（`calcCharacterStats`）：

| 屬性 | 公式 | 主要影響的五行 |
|------|------|---------------|
| **HP** | `100 + 等級×10 + 木×3.0` | 木 |
| **攻擊** | `10 + 等級×2 + 火×1.5` | 火 |
| **防禦** | `10 + 等級×1.5 + 土×1.5` | 土 |
| **速度** | `5 + 等級×0.5 + 金×0.8` | 金 |
| **魔攻** | `10 + 等級×2 + 水×1.5` | 水 |
| **MP** | `50 + 等級×5 + 水×1.5` | 水 |

## 七、發現的問題與優化建議

### 問題 1：怪物技能是純裝飾，不影響戰鬥

**現狀**：怪物技能只是隨機選一個名稱顯示，傷害固定用 `monsterBaseDmg × randFloat(0.8, 1.2)`。`gameMonsterSkillCatalog` 中定義的 `powerPercent`、`skillType`、`additionalEffect`、`aiCondition` 完全沒有被使用。

**建議**：讓戰鬥引擎從資料庫讀取怪物的技能資料，根據 `skillType` 和 `powerPercent` 計算實際傷害，並使用 `aiCondition` 做智慧選擇。具體做法：

- 怪物攻擊技能：`monsterBaseDmg × (powerPercent / 100) × randFloat(0.85, 1.15)`
- 怪物治癒技能：`monsterMaxHp × 0.15 × (powerPercent / 100)`（HP 低於某閾值時使用）
- 怪物 Buff/Debuff：提供臨時屬性加成或降低對手屬性
- `aiCondition.hpBelow`：當怪物 HP 低於此百分比時優先使用該技能
- `aiCondition.priority`：數值越高越優先選擇

### 問題 2：靜態怪物表與資料庫怪物圖鑑分離

**現狀**：戰鬥用 `shared/monsters.ts` 的硬編碼數據，後台 CMS 管理的是 `gameMonsterCatalog` 資料庫。兩套數據互不影響。

**建議**：將戰鬥引擎改為從資料庫讀取怪物數據。在每次 Tick 開始時（或使用快取），從 `gameMonsterCatalog` 載入怪物列表，並 JOIN `gameMonsterSkillCatalog` 取得技能詳情。這樣後台的修改就能即時反映到戰鬥中。

### 問題 3：魔法係數 A 沒有根據實際使用技能的屬性計算

**現狀**：`skillElement` 固定為 `agentElement`（角色主屬性），不管玩家實際用了什麼屬性的技能。

**建議**：在每回合選擇技能後，根據該技能的實際屬性重新計算 `spiritCoeffA`。這樣裝備不同屬性技能才有意義。

### 問題 4：所有技能冷卻固定 2 回合

**現狀**：不管技能本身定義的 `cooldown` 是多少，使用後一律冷卻 2 回合。

**建議**：使用 `gameSkillCatalog` 中定義的 `cooldown` 值，讓不同技能有不同的冷卻時間。強力技能冷卻長，弱技能冷卻短，增加策略深度。

### 問題 5：怪物沒有 MP 和冷卻系統

**現狀**：怪物可以無限使用任何技能，沒有 MP 消耗和冷卻限制。

**建議**：為怪物加入簡化版的 MP 系統。怪物的 MP 可以根據等級和稀有度自動計算（例如 `baseMp = 30 + level × 2`），技能消耗 MP，MP 不足時只能普攻。這會讓 Boss 戰更有節奏感。

### 問題 6：附加效果（中毒、灼燒、眩暈等）未實作

**現狀**：`additionalEffect` 欄位在 AI 生成技能時有填入（如 `{type: "poison", chance: 20, duration: 3, value: 5}`），但戰鬥引擎完全沒有處理。

**建議**：實作狀態效果系統：
- **中毒（poison）**：每回合損失 `value` 點 HP，持續 `duration` 回合
- **灼燒（burn）**：每回合損失 `value` 點 HP，持續 `duration` 回合
- **冰凍（freeze）**：`chance%` 機率跳過下一回合
- **眩暈（stun）**：`chance%` 機率跳過下一回合
- **減速（slow）**：降低速度，可能改變先手順序

### 問題 7：AI 生成的怪物技能品質缺乏驗證

**現狀**：AI 生成技能後直接寫入資料庫，沒有數值平衡檢查。

**建議**：加入數值校驗層：
- `powerPercent` 不超過稀有度上限（common: 120, rare: 150, epic: 200, legendary: 250）
- `mpCost` 不超過 30
- `cooldown` 不超過 5
- 每隻怪物至少有 1 個 attack 類型技能

## 八、完整傷害計算流程圖

以下是一次完整攻擊的傷害計算流程：

```
玩家攻擊力（attack）
    │
    ├── × 五行倍率（0.7 / 1.0 / 1.2 / 1.5）
    ├── × 魔法係數A（0.1 ~ 1.1）
    ├── × 種族倍率（0.85 / 1.0 / 1.2）
    ├── − 怪物防禦 × 0.5
    │
    ▼ 基礎傷害（最低 1）
    │
    ├── × Combo 傷害倍率（1.0 + Combo加成 + 命格加成）
    ├── × 覺醒倍率（1.0 / 1.3 / 1.6 / 2.0）
    │   ↑ 這兩步在 processCombatEvent 中，
    │     attack 已經乘過 comboResult.damageMultiplier
    │
    ├── × 技能倍率（skill.damageMultiplier）
    ├── × 隨機浮動（0.8 ~ 1.2）
    │
    ▼ 原始傷害
    │
    ├── 閃避判定 → 命中？
    │   └── 閃避機率 = min(0.2, 怪物速度/(怪物速度+玩家速度+15))
    ├── 格擋判定 → 格擋？→ 傷害 ×0.5
    │   └── 格擋機率 = 10%
    ├── 暴擊判定 → 暴擊？→ 傷害 ×1.8
    │   └── 暴擊機率 = 10%
    │
    ▼ 最終傷害
```

## 九、數值模擬範例

假設一個 Lv.20 火屬性角色（火五行 40）攻擊一個 Lv.15 金屬性怪物（defense 35）：

| 步驟 | 計算 | 結果 |
|------|------|------|
| 基礎攻擊力 | 10 + 20×2 + 40×1.5 | **110** |
| 五行倍率 | 火剋金 | **×1.5** |
| 敵方抗性 | max(5, round(35 / 1.5)) | **23** |
| 精神比例 | (40 / 23) × 100 | **174** |
| 魔法係數A | > 119 | **×1.1** |
| 種族倍率 | 人型系 vs 金屬系（人型剋金屬） | **×1.2** |
| 基礎傷害 | max(1, round(110 × 1.5 × 1.1 × 1.2 − 35 × 0.5)) | **200** |
| Combo（假設火系Combo + 命格） | 1.0 + 0.15 + 0.20 = 1.35 | 已乘入 attack |
| 覺醒（假設奧義） | ×1.6 | 已乘入 attack |
| 技能倍率（假設 150%） | ×1.5 | **×1.5** |
| 隨機浮動 | ×1.0（取中間值） | **×1.0** |
| **最終傷害（無閃避/格擋/暴擊）** | 200 × 1.5 × 1.0 | **≈300** |

## 十、總結與下一步行動建議

目前戰鬥系統的**玩家端技能 AI 判定**已經相當完善（五層優先級策略），但**怪物端**的技能系統基本上是空殼。最有價值的優化方向，按優先級排列：

1. **讓怪物技能真正生效**（影響最大）— 將 `gameMonsterSkillCatalog` 的數據接入戰鬥引擎
2. **統一怪物數據源**（基礎建設）— 從資料庫讀取怪物，取代靜態陣列
3. **實作附加效果系統**（戰略深度）— 中毒、灼燒、眩暈等狀態效果
4. **修正魔法係數 A 的技能屬性判定**（數值正確性）— 根據實際技能屬性計算
5. **使用技能自身的冷卻值**（細節完善）— 取代固定 2 回合冷卻

這些優化完成後，怪物工坊中 AI 生成的技能才能真正在戰鬥中發揮作用，整個戰略系統的深度也會大幅提升。
