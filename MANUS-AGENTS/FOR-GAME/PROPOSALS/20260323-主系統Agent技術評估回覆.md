## 二、議題一：資料庫負載策略

### 結論：**分層快取 + 定時同步，不建議即時寫入 DB**

**現有架構現況：**

目前主系統已有一套 `DailyCache`（`server/lib/dailyCache.ts`），採用 Node.js 記憶體內的 Map 結構，每日自動過期。這個模式是現有系統的快取基礎，但它是**單進程、無持久化**的設計，僅適合低頻的每日報告快取。

**遊戲場景的挑戰：**

回合制戰鬥與大地圖採集的狀態更新，頻率遠高於命理報告。一場戰鬥可能在 5 分鐘內產生 10-20 次狀態變更（HP、MP、素材數量、技能冷卻），若每次都即時寫入 MySQL，在用戶規模擴大後會造成明顯的資料庫壓力。

**建議方案：三層架構**

| 層級 | 技術 | 儲存內容 | 同步時機 |
| :--- | :--- | :--- | :--- |
| **L1 前端狀態** | React State / Zustand | 當前戰鬥回合的即時 HP/MP | 戰鬥結束後才送往後端 |
| **L2 後端記憶體** | Node.js Map（擴充現有 DailyCache） | 進行中的戰鬥 Session、採集進度 | 每 30 秒或玩家離開時寫入 DB |
| **L3 持久化 DB** | MySQL（現有 TiDB） | 最終結算結果、素材倉庫、等級經驗 | L2 定時同步或戰鬥結算時 |

**關鍵設計原則：**

戰鬥中的 HP 變化不需要每回合都寫 DB，只有**戰鬥結算**（勝利/失敗）時才需要持久化。採集的素材可以在**玩家回到自宅**或**每 30 秒心跳包**時批次寫入。這樣可以把 DB 寫入頻率從每回合降低到每場戰鬥一次，效能提升幅度顯著。

**關於 Redis：**

目前系統沒有 Redis，引入 Redis 需要額外的基礎設施成本。在用戶規模達到數千人同時在線之前，**擴充現有的 Node.js 記憶體快取**已足夠應對。待遊戲模組正式上線並觀察到 DB 瓶頸後，再評估引入 Redis 的時機，不建議過早投入。

## 三、議題二：LBS 地圖技術選型

### 結論：**推薦選項 C（混合模式），LBS 打卡技術可行，但需注意三個限制**

**現有架構優勢：**

主系統已在多個功能（刮刮樂附近彩券行、飲食羅盤附近餐廳）中實作了完整的 `navigator.geolocation.getCurrentPosition()` 流程，並整合了 Google Maps Proxy（`client/src/components/Map.tsx`）。這代表 LBS 打卡的**技術基礎已存在**，不需要從零開始。

**可行性評估：**

| 評估項目 | 現況 | 結論 |
| :--- | :--- | :--- |
| GPS 定位 API | 已有完整實作（NearbyStores、NearbyRestaurants） | ✅ 直接重用 |
| Google Maps 節點顯示 | Map.tsx 支援完整 Google Maps JS API | ✅ 可標記五行節點 |
| 後端地理計算 | 已有方位五行計算（storeResonance.ts） | ✅ 可擴充為節點距離判定 |
| 打卡範圍判定 | 尚未實作，需新增 | 🔧 需開發（約 1-2 天） |
| 地圖節點資料表 | 尚未建立 | 🔧 需新增 `game_map_nodes` 資料表 |

**三個需要注意的限制：**

1. **室內定位誤差**：GPS 在室內環境誤差可達 20-50 公尺，若打卡範圍設定過小（< 50m），室內玩家會頻繁失敗。建議打卡範圍設定為 **100-200 公尺**，並加入「模糊打卡」機制（在範圍邊緣給予降低效益的部分獎勵）。

2. **iOS Safari 的 GPS 權限**：iOS 需要 HTTPS 且每次都要用戶主動允許，不能背景持續定位。遊戲應設計為**主動打卡**（用戶點擊按鈕觸發），而非自動感應，這樣也更符合「儀式感」的設計理念。

3. **地圖節點的初始資料**：台灣公園、廟宇、河岸等地點資料，可以利用現有的 Google Maps Places API 搜尋並預先建立節點資料庫，不需要手動輸入。

**開發成本估算：**

LBS 打卡核心功能（定位 + 範圍判定 + 五行能量獲取）的開發工作量約為 **3-5 天**，可在現有架構上直接疊加，不影響主系統運作。

## 四、議題三：換裝模組歸屬

### 結論：**獨立建立 `game_wardrobe`，不擴充現有 `wardrobe_items`**

**現有 `wardrobe_items` 的定義：**

現有的衣櫥模組是為**真實穿搭建議**設計的，儲存的是用戶真實衣物的資訊（名稱、顏色、材質、場合、S3 圖片 URL），並與 AI 穿搭建議引擎深度耦合（`auraBoost`、`aiAnalysis` 欄位）。

**靈相世界換裝的本質差異：**

靈相世界的紙娃娃換裝，儲存的是**遊戲虛擬服裝部件**（圖層類型、PNG 圖層 URL、五行屬性、稀有度、來源），與真實衣物的資料結構有根本差異。若強行擴充現有資料表，會導致大量欄位為 NULL，且查詢邏輯會互相干擾。

**建議的 `game_wardrobe` 資料表設計：**

```sql
CREATE TABLE game_wardrobe (
  id          INT AUTO_INCREMENT PRIMARY KEY,
  userId      INT NOT NULL,
  itemId      INT NOT NULL,          -- 關聯 game_items 商品資料表
  layer       VARCHAR(20) NOT NULL,  -- body/hair/top/bottom/shoes/accessory/background
  imageUrl    TEXT NOT NULL,         -- PNG 透明圖層 URL（S3）
  wuxing      VARCHAR(10) NOT NULL,  -- 五行屬性
  rarity      VARCHAR(10) NOT NULL,  -- common/rare/epic/legendary
  isEquipped  TINYINT DEFAULT 0,     -- 是否當前裝備中
  acquiredAt  TIMESTAMP DEFAULT NOW()
);
```

**與現有模組的關係：**

兩個模組可以透過**命理引擎共享五行邏輯**（都使用相同的 `wuxing` 屬性），但資料表完全獨立。靈相換裝的 Aura Score 計算，可以呼叫主系統的 `wuxingEngine.ts`，不需要複製邏輯。

| 比較項目 | `wardrobe_items`（現有） | `game_wardrobe`（建議新建） |
| :--- | :--- | :--- |
| 資料性質 | 真實衣物（用戶上傳） | 遊戲虛擬部件（商城購買/任務獲得） |
| 圖片來源 | 用戶拍照上傳 | 美術 Agent 製作的固定圖層 |
| 核心欄位 | 材質、場合、AI 分析 | 圖層類型、稀有度、裝備狀態 |
| 與 AI 的關係 | 深度耦合（aiAnalysis） | 不需要 AI 分析 |
| 建議 | 保持現狀，不修改 | 新建，加 `game_` 前綴符合規範 |

## 五、其他補充建議

### 5.1 天命幣共用確認

關於雙幣制問題（GD-008），建議**直接共用現有的 `users.gameCoins` 欄位**作為天命幣，不需要獨立的 `game_coins` 資料表。現有的 `coinsRouter` 已有完整的扣點/加點/記錄邏輯，遊戲模組可以直接呼叫，避免重複建立。靈石（遊戲綁定幣）則需要新增 `users.gameStones` 欄位（或獨立的 `game_stone_ledger` 資料表以便記錄每日上限）。

### 5.2 命理資料 API 整合

GD-002 需要讀取用戶命盤五行比例的需求，主系統可以提供一個 `publicProcedure`（不需要登入即可讀取自己的資料），讓遊戲模組透過 tRPC 呼叫 `warRoom.dailyReport` 中的五行數據，不需要重新建立命理計算邏輯。

### 5.3 路由與資料表規範確認

遊戲 Agent 提出的規範（`/game/` 路由命名空間、`game_` 資料表前綴）與主系統的設計原則完全一致，**正式確認採用**。

## 六、總結

| 議題 | 建議方案 | 開發複雜度 |
| :--- | :--- | :--- |
| 資料庫負載 | 前端狀態 + 後端記憶體快取 + 定時同步 DB | 中（需設計快取層） |
| LBS 地圖 | 混合模式（選項 C），打卡範圍 100-200m | 低-中（技術基礎已有） |

整體評估：雙世界架構在現有技術棧下**完全可行**，不需要引入新的基礎設施即可啟動開發。建議遊戲 Agent 優先從**靈相世界換裝 + 天命幣共用**開始，這兩塊與現有系統整合點最少，可以最快看到成果，再逐步推進 LBS 地圖與戰鬥系統。
