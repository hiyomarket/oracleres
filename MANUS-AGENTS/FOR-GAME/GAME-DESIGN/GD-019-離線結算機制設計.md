## 一、設計動機：為什麼需要離線結算

### 1.1 伺服器壓力問題

目前沙盒採用「全局同步 Tick」架構：每 5 秒觸發一次計時器，對所有在線玩家同時結算戰鬥、移動、採集，並寫入資料庫。當玩家人數增加時，這個架構會產生嚴重的「寫入峰值」問題。

以下是不同玩家規模下的資料庫寫入壓力估算：

| 玩家人數 | 每次 Tick 的 DB 寫入筆數 | 每分鐘總寫入量 | 風險等級 |
|---------|----------------------|-------------|--------|
| 100 人 | ~300 筆（角色狀態 + 事件日誌） | ~3,600 筆 | 低風險 |
| 500 人 | ~1,500 筆 | ~18,000 筆 | 中風險 |
| 1,000 人 | ~3,000 筆 | ~36,000 筆 | **高風險** |
| 5,000 人 | ~15,000 筆 | ~180,000 筆 | **崩潰風險** |

離線結算機制將這個問題從根本上解決：**伺服器在玩家不活躍時完全不做任何計算**，只有玩家登入的瞬間才進行一次性的離線期間結算，將瞬間峰值轉變為分散的登入時計算。

### 1.2 遊戲體驗問題

即時 Tick 架構還有一個遊戲設計上的缺陷：玩家必須持續盯著手機才能感受到遊戲的進展，這與放置型遊戲「掛機也能玩」的核心承諾相違背。

離線結算機制反而創造了放置遊戲最核心的樂趣——**「開箱感」**：每次打開 App，玩家都會看到「我不在的這段時間，旅人發生了什麼事」，就像拆開一個驚喜包裹。這種感覺比即時更令人期待，也是 AFK Arena、Melvor Idle、Idle Champions 等放置遊戲長期留存率高的主要原因。

## 二、核心架構：雙模式 Tick 系統

### 2.1 架構概覽

離線結算機制採用「雙模式」設計，根據玩家的活躍狀態自動切換：

```
玩家狀態判斷
│
├── 活躍狀態（最後操作 < 5 分鐘前）
│   └── 即時 Tick：每 60 秒結算一次（降低頻率）
│       ├── 結算戰鬥/採集/移動
│       ├── 更新角色狀態
│       └── 推送事件到前端（SSE 或輪詢）
│
└── 離線狀態（最後操作 > 5 分鐘前）
    └── 暫停所有 Tick，記錄「離線開始時間」
        │
        └── 玩家下次登入時
            └── 離線結算：一次性計算離線期間所有事件
                ├── 計算離線時長（上限 12 小時）
                ├── 模擬離線期間的行動序列
                ├── 生成「離線報告」（文字摘要）
                └── 呈現給玩家（開箱感 UI）
```

### 2.2 關鍵設計決策

**決策 1：離線計算上限為 12 小時**

離線結算不是無限累積的。超過 12 小時的離線時間，超出部分不計算。這個設計有三個目的：

一是防止玩家長時間掛機獲得過多資源，破壞遊戲平衡；二是讓每日登入有意義（每天至少登入一次才能最大化收益）；三是降低單次結算的計算量，確保登入時的計算在 1 秒內完成。

**決策 2：離線效率為即時效率的 60%**

玩家離線期間，旅人的行動效率降低至即時狀態的 60%。這個設計確保主動玩家（使用體力值干預策略）的收益高於純掛機玩家，同時讓掛機玩家也有足夠的基礎收益維持遊戲進度。

**決策 3：離線期間使用「最後設定的策略」**

玩家離線時，旅人繼續按照玩家最後設定的行動策略（探索/刷怪/採集/休息）執行。這讓玩家在離線前的策略選擇變得有意義——出門前設定好策略，回來看結果。

**決策 4：體力值在離線期間正常恢復，但不消耗**

離線期間，旅人的體力值按正常速率恢復（每 30 分鐘 +1 活躍值），但不消耗體力值進行行動。體力值是玩家主動干預的資源，離線期間的自動行動使用另一套「基礎行動點」計算（見第三節）。

## 三、離線結算演算法

### 3.1 觸發條件

以下任一條件觸發離線結算：

1. 玩家登入時，`lastActiveAt` 距今超過 5 分鐘
2. 玩家手動點擊「結算離線收益」按鈕（可在任何時候觸發）

### 3.2 計算流程

```
輸入：
  - offlineStart: 離線開始時間（Unix timestamp ms）
  - offlineEnd: 當前時間
  - agent: 旅人當前狀態（位置、HP、策略、等級、五行屬性）
  - strategy: 最後設定的行動策略

計算步驟：

Step 1：計算有效離線時長
  rawDuration = offlineEnd - offlineStart
  effectiveDuration = min(rawDuration, 12 * 60 * 60 * 1000)  // 上限 12 小時

Step 2：計算基礎行動次數
  baseActionInterval = 根據策略決定（見下表）
  totalActions = floor(effectiveDuration / baseActionInterval)
  
  策略與行動間隔對照表：
  ┌──────────┬──────────────┬─────────────────────────────────┐
  │ 策略     │ 行動間隔     │ 說明                            │
  ├──────────┼──────────────┼─────────────────────────────────┤
  │ 探索     │ 8 分鐘/次    │ 移動 + 遭遇，效率均衡           │
  │ 刷怪     │ 5 分鐘/次    │ 戰鬥頻繁，EXP/金幣較多         │
  │ 採集     │ 10 分鐘/次   │ 採集素材，較少戰鬥              │
  │ 休息     │ 15 分鐘/次   │ HP/MP 恢復，行動次數最少        │
  └──────────┴──────────────┴─────────────────────────────────┘

Step 3：套用離線效率係數
  effectiveActions = floor(totalActions * 0.6)
  // 離線效率 60%，確保主動玩家優勢

Step 4：模擬行動序列（批次計算）
  對每一次 effectiveActions 執行：
    a. 判斷行動類型（依策略機率分配）
    b. 計算戰鬥/採集/移動結果
    c. 套用五行加成/相剋計算
    d. 累積：EXP、金幣、素材、HP 消耗
    e. 若 HP 降至 0：進入「昏迷狀態」，後續行動暫停，改為恢復
    f. 每 20 次行動：有 5% 機率觸發 Roguelike 事件（存入補簽到佇列）

Step 5：計算體力值恢復
  staminaRecovered = floor(effectiveDuration / (30 * 60 * 1000))
  // 每 30 分鐘恢復 1 點活躍值，上限為 maxStamina

Step 6：生成離線報告
  summary = {
    duration: 有效離線時長（小時分鐘）,
    actionsCount: 實際行動次數,
    battlesWon: 勝利場次,
    battlesLost: 失敗場次,
    expGained: 獲得 EXP,
    goldGained: 獲得金幣,
    itemsFound: 找到的道具清單,
    levelUps: 升級次數（0 或多次）,
    roguelikeEvents: 待處理的肉鴿事件數量,
    currentLocation: 旅人當前位置,
    hpStatus: 旅人 HP 狀態（正常/昏迷）
  }

輸出：
  - 更新後的 agent 狀態
  - offlineReport（供前端顯示）
  - roguelikeEventQueue（待玩家處理的事件清單）
```

### 3.3 五行加成在離線結算中的應用

離線結算中，五行加成的計算方式與即時 Tick 相同，但採用「批次平均」而非逐次計算，以降低計算量：

```
// 離線戰鬥結果計算（批次版）
offlineBattleResult = {
  winRate: 基礎勝率 * 五行相剋係數 * 裝備加成,
  avgExpPerBattle: 基礎 EXP * 五行生剋係數,
  avgGoldPerBattle: 基礎金幣 * 地圖財富係數,
  avgHpLossPerBattle: 基礎傷害 * (1 - 防禦係數)
}

// 批次計算 N 場戰鬥的結果
totalExp = N * offlineBattleResult.avgExpPerBattle * 0.6  // 離線效率
totalGold = N * offlineBattleResult.avgGoldPerBattle * 0.6
totalHpLoss = N * offlineBattleResult.avgHpLossPerBattle
```

這個批次計算方式確保了：無論離線了 1 小時還是 12 小時，計算時間都在毫秒級別，不會造成登入時的延遲。

### 4.2 文字敘事風格

離線報告的文字應採用「說書人」風格，而不是乾燥的數字統計。以下是範例對照：

**不好的寫法（數字堆疊）**：
> 離線 8 小時。戰鬥 16 次，勝率 87.5%。獲得 EXP 1240，金幣 380，素材 12 件。

**好的寫法（說書人風格）**：
> 你離開的這段時間，旅人獨自穿越了中央山脈的重重霧林。在阿里山的神木群間，他遭遇了一隻「霧靈熊」的突襲，雖然身受輕傷，卻從熊爪下搶到了一枚罕見的「木靈晶核」。臨近傍晚，一位身份不明的流浪商人在神木廣場支起了攤位——他似乎在等待什麼人。

這種敘事風格讓玩家感覺旅人是真實存在的角色，而不只是一組數字在自動更新。

### 4.3 稀有道具的特殊呈現

若離線期間獲得了品質為「玄（藍）」以上的道具，在離線報告中應有特殊動畫效果：

- 道具圖示從畫面底部飛入
- 顯示品質光效（藍色/紫色/橙色光暈）
- 播放短暫的音效（可選）
- 道具名稱以對應品質顏色顯示

這個設計讓稀有掉落的「驚喜感」最大化，是玩家持續登入的重要動機。

### 4.4 昏迷狀態的特殊處理

若旅人在離線期間 HP 歸零（昏迷），離線報告應有特殊提示：

```
⚠️ 旅人在「玉山主峰」遭遇強敵，力竭昏迷。
   昏迷後的 3 小時 15 分鐘，旅人在原地休養，
   HP 已自然恢復至 45%。
   
   [立即治療（消耗 1 靈石）]  [繼續休養]
```

昏迷狀態不是懲罰，而是一個敘事節點，讓玩家感受到旅人的脆弱性，同時提供消費靈石的自然場景。

## 五、體力值與離線結算的關係

### 5.1 雙軌行動資源

離線結算引入了「雙軌行動資源」的概念，明確區分主動玩家和掛機玩家的資源來源：

| 資源類型 | 來源 | 用途 | 離線期間 |
|---------|------|------|---------|
| **活躍值**（體力值） | 每 30 分鐘 +1，上限 10 | 玩家主動觸發的高效行動（神蹟、策略切換、特殊任務） | 正常恢復，不消耗 |
| **基礎行動點** | 離線期間自動累積 | 離線結算的自動行動 | 按策略間隔自動消耗 |

這個設計確保：
- 掛機玩家有基礎收益，不會因為不玩而完全落後
- 主動玩家的活躍值帶來額外的高效行動，形成明確的差異化優勢
- 體力值不會在離線期間被「浪費」，玩家回來時有滿格的體力值可以使用

### 5.2 體力值的最佳使用時機

離線結算機制確立了一個清晰的「最佳遊玩節奏」：

```
玩家最佳遊玩節奏（每日）：

早上登入（8:00）
  → 查看離線報告（8 小時的收益）
  → 處理 Roguelike 事件（若有）
  → 使用體力值執行 2-3 次高效行動（神蹟/特殊任務）
  → 設定今日策略後離線（約 10-15 分鐘遊玩時間）

中午登入（12:00）
  → 查看 4 小時離線報告
  → 使用恢復的 4 點活躍值
  → 設定下午策略後離線（約 5-10 分鐘）

晚上登入（20:00）
  → 查看 8 小時離線報告
  → 使用滿格的活躍值進行最重要的行動
  → 設定隔夜策略（採集/探索）後離線（約 15-20 分鐘）

每日總遊玩時間：30-45 分鐘
每日總遊玩體驗：3 次「開箱感」+ 策略選擇樂趣
```

這個節奏設計讓遊戲對忙碌的成年玩家非常友善，同時每次登入都有足夠的內容讓玩家感到「有事可做」。

## 七、伺服器架構優化建議

### 7.1 Tick 引擎改造方向

從「全局同步 Tick」改為「按需觸發」架構：

```
舊架構（全局同步）：
  setInterval(() => {
    allActivePlayers.forEach(player => tickPlayer(player))  // 同時處理所有玩家
  }, 5000)

新架構（按需觸發）：
  // 玩家登入時：觸發離線結算
  onPlayerLogin(player) {
    if (isOffline(player)) {
      result = calculateOfflineProgress(player)
      showOfflineReport(player, result)
    }
    markPlayerActive(player)
  }
  
  // 玩家活躍時：個別計時器，錯開觸發
  onPlayerActive(player) {
    scheduleNextTick(player, randomOffset(55, 65))  // 55-65 秒隨機間隔，避免同步
  }
  
  // 玩家離線時：停止個別計時器
  onPlayerInactive(player) {
    cancelTick(player)
    recordOfflineStart(player)
  }
```

### 7.2 預期的效能改善

採用新架構後，伺服器壓力的改善預估：

| 指標 | 舊架構（1000 人） | 新架構（1000 人） | 改善幅度 |
|------|----------------|----------------|---------|
| 每分鐘 DB 寫入 | ~36,000 筆 | ~600 筆（假設 10% 同時在線） | **-98%** |
| 峰值 CPU 使用率 | 高（每 5 秒峰值） | 低（分散觸發） | **-85%** |
| 記憶體佔用 | 高（維護所有玩家狀態） | 低（只維護在線玩家） | **-70%** |
| 登入時延遲 | 無 | 約 200-500ms（離線結算計算） | 可接受 |

## 八、Roguelike 事件與離線結算的整合

### 8.1 事件觸發機制

離線結算期間，每 20 次行動有 5% 機率觸發一個 Roguelike 事件。觸發的事件不會立即結算，而是存入「待處理事件佇列」，等玩家登入後手動處理。

這個設計的目的是：讓 Roguelike 事件保持「玩家主動參與」的性質，而不是在離線期間自動結算（那樣就失去了選擇的樂趣）。

### 8.2 事件佇列上限

待處理事件佇列最多存放 **5 個**事件。超過 5 個時，新觸發的事件會替換最舊的一個（並給予玩家一個「錯過了一個事件」的提示）。這個設計防止長時間不登入的玩家積累過多未處理事件，同時創造「定期登入處理事件」的動機。

### 8.3 事件過期機制

每個 Roguelike 事件有 **72 小時**的有效期。超過有效期後，事件自動消失，玩家無法再處理。部分事件（如「流浪商人」）有更短的有效期（24 小時），在離線報告中會特別標注「限時！」提示。

## 九、尚待補完的部分（10%）

以下內容需要在數值平衡測試後補充：
