## 四、P1 補充：命格五行影響掉落的深化設計

### 現有設計的問題

GD-015B 的五行掉落加成只考慮「攻擊屬性 vs 怪物屬性」的剋制關係（剋制 ×1.5、相生 ×0.8）。這個設計是正確的，但缺少一個更重要的維度：**玩家的命格五行比例應該影響特定屬性裝備的掉落機率**。

### 補充設計：命格共鳴掉落加成

玩家的命格五行比例越高，對應屬性的裝備掉落率有額外加成。這個設計的核心邏輯是：「天命引導旅人走向與自身命格相符的緣分」。

#### 命格共鳴加成公式

```
命格共鳴加成 = 1 + (玩家對應五行比例 - 20%) × 0.5

上限：×1.5（五行比例 100% 時）
下限：×0.5（五行比例 0% 時）
基準線：五行比例 20% 時，加成為 ×1.0（無加成）
```

**計算範例：**

| 玩家命格 | 木屬性比例 | 木系裝備命格共鳴加成 |
|:---|:---:|:---:|
| 純木命（木 80%）| 80% | 1 + (80-20)×0.5% = **×1.30** |
| 均衡命（木 20%）| 20% | 1 + (20-20)×0.5% = **×1.00** |
| 無木命（木 5%）| 5% | 1 + (5-20)×0.5% = **×0.925** |

> 命格共鳴加成的影響幅度刻意設計得比五行剋制加成（×1.5）小，確保「用正確屬性攻擊」的策略性仍然重要，命格只是錦上添花而非決定性因素。

#### 最終掉落率完整公式

```
最終掉落率 = 基礎掉落率 × 屬性加成 × 命格共鳴加成 × 幸運加成 × 任務加成

其中：
- 基礎掉落率：怪物種類決定（見 GD-015B 第三章）
- 屬性加成：攻擊屬性 vs 怪物屬性的五行剋制關係（×0.8 ~ ×1.5）
- 命格共鳴加成：玩家命格五行比例對應掉落裝備屬性（×0.925 ~ ×1.30）
- 幸運加成：玩家幸運值 / 1000（最高 +50%，對應金屬性高的玩家）
- 任務加成：特定任務期間 ×1.2
```

**實際案例（木命玩家用金屬武器打木系怪，掉落木系裝備）：**

```
基礎掉落率 = 8%（初階怪物）
屬性加成 = ×1.5（金剋木）
命格共鳴加成 = ×1.30（木屬性 80%）
幸運加成 = ×1.0（假設無加成）
任務加成 = ×1.0（假設無任務）

最終掉落率 = 8% × 1.5 × 1.30 × 1.0 × 1.0 = 15.6%
```

這個設計讓木命玩家在木系區域用金屬武器打怪，掉落率接近原始分析報告的 15~20%，符合玩家的直覺預期，同時又有命格差異化的體驗。

## 五、P1 補充：放置遊戲的掉落結算時機

### 問題

GD-015B 的掉落系統是為「即時戰鬥」設計的，但虛相世界是放置型文字遊戲，戰鬥在 Tick 引擎中自動進行，需要明確定義掉落結算的時機。

### 掉落結算流程（配合 GD-019 離線結算）

```
在線模式（旅人活躍中）：
每次 Tick 戰鬥結算 → 立即計算掉落 → 寫入 agent_inventory → 推送事件日誌

離線模式（玩家不在線）：
離線期間的每次虛擬戰鬥 → 記錄戰鬥結果（含掉落物）→ 玩家登入時一次性展示
```

### 掉落物的存放規則

| 情況 | 處理方式 |
|:---|:---|
| 背包未滿（< 50 格）| 直接放入背包 |
| 背包已滿 | 放入「待領取倉庫」，保留 7 天 |
| 離線掉落 | 放入「離線報告」，玩家確認後入庫 |
| 傳說裝備掉落 | 強制推送通知，不受背包限制 |

### 背包與倉庫規格

```typescript
// 背包規格
const INVENTORY_CONFIG = {
  maxSlots: 50,          // 基礎背包格數
  expandSlots: 20,       // 每次擴充格數（需消耗靈石）
  maxExpands: 5,         // 最多擴充 5 次（上限 150 格）
  offlineReportExpiry: 7 * 24 * 60 * 60 * 1000, // 離線報告保留 7 天
};
```

## 六、P0 補充：DB Schema 完整定義

以下是裝備掉落系統需要的完整資料庫表格，供主系統 Agent 實作時參考。

### 6.1 裝備主表（equipment_templates）

```sql
CREATE TABLE equipment_templates (
  id          VARCHAR(20) PRIMARY KEY,  -- E_W001, E_F001 等
  name        VARCHAR(50) NOT NULL,
  element     ENUM('wood','fire','earth','metal','water') NOT NULL,
  tier        ENUM('basic','mid','high','legendary') NOT NULL,
  slot        ENUM('weapon','helmet','armor','boots','accessory') NOT NULL,
  level_req   INT NOT NULL DEFAULT 1,
  
  -- 基礎屬性加成
  hp_bonus    INT DEFAULT 0,
  atk_bonus   INT DEFAULT 0,
  def_bonus   INT DEFAULT 0,
  spd_bonus   INT DEFAULT 0,
  matk_bonus  INT DEFAULT 0,
  mp_bonus    INT DEFAULT 0,
  
  -- 特殊詞條（JSON 格式，最多 3 個）
  affixes     JSON,
  
  -- 套裝資訊
  set_id      VARCHAR(20),  -- 例如 'SET_WOOD_LEGENDARY'
  set_piece   INT,          -- 套裝第幾件（1~5）
  
  -- 商店定價
  shop_price  INT,          -- NULL 表示不可購買
  npc_sell_price INT,       -- 玩家賣給 NPC 的價格
  
  -- 是否可交易
  tradeable   BOOLEAN DEFAULT TRUE,
  
  created_at  BIGINT NOT NULL
);
```

### 6.2 怪物掉落表（monster_drop_tables）

```sql
CREATE TABLE monster_drop_tables (
  id            INT AUTO_INCREMENT PRIMARY KEY,
  monster_id    VARCHAR(20) NOT NULL,  -- M_W001 等
  equipment_id  VARCHAR(20) NOT NULL,  -- E_W001 等
  drop_rate     DECIMAL(5,4) NOT NULL, -- 0.0700 = 7%
  is_penetrate  BOOLEAN DEFAULT FALSE, -- 是否為滲透掉落（跨階）
  
  INDEX idx_monster (monster_id)
);
```

### 6.3 玩家背包表（agent_inventory）

```sql
CREATE TABLE agent_inventory (
  id            INT AUTO_INCREMENT PRIMARY KEY,
  agent_id      INT NOT NULL,
  equipment_id  VARCHAR(20) NOT NULL,
  
  -- 個體詞條（每件裝備的隨機詞條，JSON）
  rolled_affixes JSON,
  
  -- 強化等級（0~10）
  enhance_level  INT DEFAULT 0,
  
  -- 是否裝備中
  is_equipped   BOOLEAN DEFAULT FALSE,
  equipped_slot ENUM('weapon','helmet','armor','boots','accessory'),
  
  -- 來源記錄
  obtained_from VARCHAR(50),  -- 'drop:M_W001', 'shop', 'craft' 等
  obtained_at   BIGINT NOT NULL,
  
  INDEX idx_agent (agent_id)
);
```

### 6.4 掉落計數表（agent_drop_counters）

## 七、TypeScript 實作建議

### 7.1 掉落計算函數

```typescript
interface DropContext {
  monsterId: string;
  monsterElement: WuxingElement;
  attackerElement: WuxingElement;
  attackerNatalStats: NatalStats;  // 命格五行比例
  luckyValue: number;              // 幸運值（0~1000）
  hasTaskBonus: boolean;
}

function calculateDropRate(
  baseRate: number,
  equipmentElement: WuxingElement,
  ctx: DropContext
): number {
  // 1. 五行剋制加成
  const elementBonus = getWuxingDropBonus(ctx.attackerElement, ctx.monsterElement);
  
  // 2. 命格共鳴加成
  const natalRatio = ctx.attackerNatalStats[equipmentElement]; // 0~100
  const natalBonus = 1 + (natalRatio - 20) * 0.005; // 每1%差距影響0.5%
  const clampedNatalBonus = Math.max(0.5, Math.min(1.5, natalBonus));
  
  // 3. 幸運加成
  const luckyBonus = 1 + ctx.luckyValue / 1000;
  
  // 4. 任務加成
  const taskBonus = ctx.hasTaskBonus ? 1.2 : 1.0;
  
  return baseRate * elementBonus * clampedNatalBonus * luckyBonus * taskBonus;
}

function getWuxingDropBonus(attacker: WuxingElement, defender: WuxingElement): number {
  if (WUXING_OVERCOME[attacker] === defender) return 1.5;  // 剋制 +50%
  if (WUXING_GENERATE[attacker] === defender) return 0.8;  // 相生 -20%
  return 1.0;
}
```

### 7.2 低保系統函數

```typescript
async function checkAndApplyPity(agentId: number, db: DB): Promise<EquipmentTier | null> {
  const counter = await db.query.agentDropCounters.findFirst({
    where: eq(agentDropCounters.agentId, agentId)
  });
  
  if (!counter) return null;
  
  // 高階保底（75 場）
  if (counter.noHighStreak >= 75) {
    await resetCounter(agentId, 'high', db);
    return 'high';
  }
  
  // 中階保底（30 場）
  if (counter.noMidStreak >= 30) {
    await resetCounter(agentId, 'mid', db);
    return 'mid';
  }
  
  // 初階保底（15 場）
  if (counter.noDropStreak >= 15) {
    await resetCounter(agentId, 'basic', db);
    return 'basic';
  }
  
  return null;
}
```

## 八、與 GD-018（等級扁平化）的整合說明

GD-018 定義了「裝備是力量的主要來源，等級只是通行證」的核心原則。本掉落系統在以下三個地方支撐這個原則：

**第一，掉落率設計讓玩家在中低等級就能獲得有意義的裝備。** 初階怪物有 7~10% 的基礎掉落率，加上命格共鳴加成最高可達 15.6%，確保新玩家不會長期空手而歸，有持續的收集進度感。

**第二，傳說裝備的每週首殺保底機制。** 這讓高等級玩家每週都有確定性的收穫，避免純靠運氣導致的挫折感，同時維持傳說裝備的稀缺性（非首殺只有 0.5%）。

**第三，套裝收集獎勵強化「收集感」而非「等級感」。** 玩家的目標是湊齊 5 件套裝，而不是升到某個等級，這讓遊戲的長期目標更有趣且多元。

## 九、與 GD-019（離線結算）的整合說明

離線結算機制對掉落系統有以下影響：

**離線掉落的呈現方式**應該比在線掉落更有「開箱感」。建議在離線報告中，把所有掉落物集中展示，並按稀有度排序（傳說 → 高階 → 中階 → 初階），讓玩家每次登入都有期待感。

**離線期間的低保計數**應該正常累積。如果玩家離線 8 小時，Tick 引擎虛擬結算了 50 場戰鬥都沒有掉落中階裝備，那麼這 50 場應該計入低保計數器，玩家下次登入後的第一場戰鬥就可能觸發保底。
