# 遊戲化模組技術決策摘要

**最後更新**：2026-03-23
**來源**：主系統 Agent 技術評估回覆 (`PROPOSALS/20260323-主系統Agent技術評估回覆.md`)

本文件記錄了天命共振遊戲化模組在初期設計階段，與主系統 Agent 協商後確認的核心技術決策。這些決策將作為後續所有 GD 文檔與實作的最高指導原則。

---

## 1. 資料庫負載與狀態同步策略

為解決回合制戰鬥與大地圖採集產生的高頻狀態更新問題，避免拖垮現有 MySQL (TiDB) 資料庫，確認採用**三層快取架構**：

- **L1 前端狀態（React State / Zustand）**：處理當前戰鬥回合的即時 HP/MP 變化與動畫表現。
- **L2 後端記憶體（Node.js Map）**：擴充現有的 `DailyCache` 機制，儲存進行中的戰鬥 Session 與採集進度。每 30 秒或玩家離開時才寫入 DB。
- **L3 持久化 DB（MySQL）**：僅在「戰鬥結算（勝利/失敗）」或「玩家回到自宅」時，才將最終結果、經驗值與掉落物一次性寫入資料庫。

> **結論**：目前不需要引入 Redis，現有架構透過分層快取即可支撐。

---

## 2. 大地圖技術選型 (LBS)

確認採用**混合模式（虛擬地圖 + 真實 GPS 打卡）**。

- **技術基礎**：主系統已有完整的 GPS 定位 API 與 Google Maps 整合基礎，可直接重用。
- **打卡範圍**：設定為 100-200 公尺，以容忍室內 GPS 誤差，並加入「模糊打卡」機制（邊緣給予部分獎勵）。
- **觸發方式**：為符合 iOS 權限限制與遊戲儀式感，採用「用戶點擊按鈕主動打卡」，而非背景自動感應。
- **節點資料**：初始地圖節點（如公園、廟宇）將透過 Google Maps Places API 自動抓取，存入新建的 `game_map_nodes` 資料表。

---

## 3. 換裝模組資料表設計

確認**獨立建立 `game_wardrobe` 資料表**，不擴充現有的 `wardrobe_items`。

- **原因**：現有模組是為真實衣物（用戶拍照上傳、AI 分析）設計，與遊戲虛擬部件（圖層類型、PNG URL、稀有度）的資料結構有根本差異。
- **五行邏輯共享**：雖然資料表獨立，但靈相換裝的 Aura Score 計算，將直接呼叫主系統現有的 `wuxingEngine.ts`，確保五行邏輯一致。

---

## 4. 經濟系統與貨幣共用

- **天命幣（儲值代幣）**：**直接共用現有主系統的 `users.gameCoins` 欄位**，不另建資料表。遊戲模組將直接呼叫現有 `coinsRouter` 的扣點/加點邏輯。
- **靈石（遊戲綁定幣）**：將在現有 `users` 資料表中新增 `gameStones` 欄位，並可能搭配獨立的 `game_stone_ledger` 資料表以記錄每日獲取上限與防刷機制。

---

## 5. 命名空間與規範確認

主系統正式確認並接受遊戲 Agent 提出的兩項核心規範：
1. **路由獨立性**：所有遊戲相關的 API 與頁面路由皆放置於 `/game/` 命名空間下。
2. **資料表前綴**：所有新增的遊戲資料表皆必須加上 `game_` 前綴。
