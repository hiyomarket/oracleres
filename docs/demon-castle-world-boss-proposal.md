# 天命共振 — 惡魔城 & 世界 Boss 系統企劃書（已確認版）

> 本文件為功能方向企劃書，設計決策已由老闆確認。待排入開發時程。

---

## 一、核心構想

本次更新將為天命共振加入兩大高階 PvE 內容：**惡魔城攻城戰**和**世界 Boss 討伐戰**。兩者共享同一套全新的**回合制手動戰鬥引擎**和**組隊模組**，是目前系統中不存在的核心模組。

### 設計原則

全服玩家共同討伐同一個目標。每個惡魔城或世界 Boss 在同一時間只允許一支隊伍進行挑戰，先打完就結束，其他隊伍需等待冷卻時間。這個設計創造了「搶攻」的緊張感和「全服協力」的社群凝聚力。

## 二、回合制手動戰鬥引擎（核心系統）

### 2.1 戰鬥操作模式

參考《魔力寶貝》的經典回合制設計。每回合 **30 秒倒數計時**，玩家可從以下四種行動中選擇一種：

| 行動類型 | 說明 | 備註 |
|----------|------|------|
| **攻擊** | 使用普通攻擊打擊目標 | 不消耗 MP |
| **技能** | 從已裝備的主動技能中選擇一個施放 | 消耗 MP，有冷卻時間 |
| **道具** | 使用背包中的消耗品（HP/MP 藥水等） | 消耗道具，不攻擊 |
| **防禦** | 本回合受到的傷害減半 | 適合血量危急時使用 |

若 30 秒內未操作，系統將根據預設邏輯自動判定行動：HP 低於 30% 時自動防禦，MP 充足時自動使用最強技能，否則自動普攻。

### 2.2 與現有系統的差異

| 項目 | 現有大地圖戰鬥 | 新回合制戰鬥 |
|------|----------------|--------------|
| 觸發方式 | Tick 自動觸發 | 玩家主動進入 |
| 操作方式 | 全自動（AI 選技能） | 手動選擇行動 |
| 回合數 | 固定 10 回合 | 無上限，直到一方倒下 |
| 多人支援 | 不支援 | 1-4 人組隊 |
| 即時性 | 非即時 | WebSocket 即時推送 |
| 狀態效果 | 無 | 灼燒/冰凍/中毒/減速等 |

### 2.3 戰鬥流程

```
玩家進入惡魔城/世界Boss → 選擇單人或組隊 → 進入戰鬥房間
    ↓
回合開始 → 每位玩家同時選擇行動（30秒倒數）
    ↓
所有玩家行動確認（或超時自動判定）→ 按速度排序結算
    ↓
判斷勝負 → 繼續下一回合 or 戰鬥結束
    ↓
結算獎勵 → 返回大地圖
```

## 三、組隊模組

| 項目 | 設計 |
|------|------|
| 隊伍上限 | 4 人一隊 |
| 建隊方式 | 隊長建立房間 → 分享邀請碼或從附近玩家邀請 |
| 隊伍通訊 | 利用現有 WebSocket 建立隊伍頻道 |
| 戰鬥行動 | 所有隊員同時下指令，30 秒後統一結算（魔力寶貝模式） |
| 斷線處理 | 斷線 30 秒內可重連，超時自動 AI 託管 |
| 獎勵分配 | 所有隊員共享經驗和金幣，裝備掉落隨機分配 |
| 挑戰限制 | 一次只能有一隊進行挑戰，其他隊伍排隊等候 |

## 四、惡魔城系統

### 4.1 五座惡魔城設定

以台灣五方位對應五行，每座惡魔城有獨立世界觀、風格難度和 AI 生成的特色設計。

| 方位 | 城名 | 五行 | 地理位置 | 領主名稱 | 城池特色與風格難度 |
|------|------|------|----------|----------|-------------------|
| 北 | **凜霜魔城** | 水 | 基隆/北海岸 | **潮淵冥王・蒼瀾** | 海霧瀰漫的港口要塞。Boss 擅長冰凍和減速，需要高速角色破解。**控制型難度**：考驗玩家的狀態解除和行動規劃 |
| 南 | **炎獄魔城** | 火 | 高雄/屏東 | **焚天魔帝・赤焱** | 熔岩地底的火山要塞。Boss 擅長灼燒和範圍傷害。**爆發型難度**：需要高治癒量和防禦輪替，否則團滅 |
| 東 | **翠嵐魔城** | 木 | 花蓮/台東 | **萬木邪神・枯榮** | 被巨藤纏繞的山谷古城。Boss 擅長吸血和持續回復。**持久型難度**：需要高輸出壓制回復，拖久必敗 |
| 西 | **金戈魔城** | 金 | 台南/嘉義 | **鑄魂鬼將・鉎鋒** | 廢棄兵器工坊改建的鐵壁堡壘。Boss 擅長高物攻和破甲。**硬碰硬難度**：純粹的數值對決，裝備不夠硬就打不動 |
| 中 | **厚土魔城** | 土 | 南投/台中 | **地裂魔尊・崩嶽** | 深埋地底的巨大地宮，入口在日月潭附近。Boss 擅長高防禦和地震技能。**策略型難度**：需要找到弱點才能破防，盲打無效 |

### 4.2 五關制設計（以凜霜魔城為例）

| 關卡 | 守衛名稱 | 定位 | 特殊機制 |
|------|----------|------|----------|
| 第一關 | 寒潮門衛 | 肉盾型 | 高 HP 低攻擊，教學性質 |
| 第二關 | 冰霜騎士 | 均衡型 | 有機率凍結玩家一回合 |
| 第三關 | 霜刃將軍 | 攻擊型 | 血量低於 50% 進入狂暴，攻擊翻倍 |
| 第四關 | 深淵大公・暗流 | 法師型 | 每 3 回合召喚水元素小兵 |
| 第五關 | 潮淵冥王・蒼瀾 | 領主 | 三階段變身，每階段技能組不同 |

### 4.3 挑戰規則

- **不限次數**：惡魔城未被攻陷前，玩家可以無限次進攻
- **一隊獨佔**：同一時間只能有一支隊伍在城內挑戰
- **撤退代價**：中途撤退從第一關重新開始
- **失敗處罰 — 惡魔城的詛咒**：挑戰失敗的玩家會受到「惡魔城詛咒」，由 AI 根據該城五行屬性發想詛咒效果（例如凜霜魔城的「冰封之咒」：30 分鐘內移動速度降低 50%、攻擊力降低 20%）
- **攻陷冷卻**：被攻陷後 3 天冷卻恢復
- **難度遞增**：每次被攻陷後 Boss 屬性 +2%（上限 +20%）

### 4.4 寶箱獎勵

| 寶箱類型 | 獲得條件 | 內容物 |
|----------|----------|--------|
| 關卡寶箱 | 每通過一關 | 1-2 件稀有(rare)以上道具/裝備 |
| 領主寶箱 | 擊敗領主 | 3-5 件史詩(epic)以上裝備/道具/技能書 |
| 領主獨特裝備 | 擊敗領主（10% 機率） | AI 即時生成的專屬傳說級裝備，帶有領主名稱 |
| 首殺獎勵 | 伺服器首次攻陷 | 獨特稱號 + 傳說裝備保底 |

## 五、世界 Boss 系統

### 5.1 設計概要

世界 Boss 固定在地圖節點上出現，是常駐的高階挑戰內容。與惡魔城不同，世界 Boss 沒有五關制，直接面對 Boss 本體和隨行小兵。

| 項目 | 設計 |
|------|------|
| 出現方式 | 固定在特定地圖節點上，常駐存在 |
| Boss 數量 | 初期 3 隻，後續可擴充 |
| 小兵機制 | Boss 身邊有 2-4 隻小兵，需要處理小兵才能對 Boss 造成完整傷害 |
| 挑戰限制 | 同一時間只能有一支隊伍挑戰 |
| 不限次數 | 未被擊敗前可無限進攻 |
| 失敗處罰 | 失敗會掉落經驗值（AI 根據 Boss 屬性發想具體掉落比例和附加效果） |
| 攻陷冷卻 | 被擊敗後進入冷卻期（時間待定） |

### 5.2 與惡魔城的差異

| 項目 | 惡魔城 | 世界 Boss |
|------|--------|-----------|
| 關卡制 | 五關制（門衛→騎士→將軍→大公→領主） | 直面 Boss + 小兵 |
| 失敗處罰 | 惡魔城詛咒（AI 發想的負面狀態） | 掉落經驗值 |
| 節點位置 | 五座固定位置（東南西北中） | 固定在特定地圖節點 |
| 難度遞增 | 每次被攻陷後 +2% 屬性 | 待定 |

## 七、技術挑戰與克服方案

### 挑戰一：即時回合制多人戰鬥（最大難點）

**問題**：現有戰鬥是「一次性結算」（`resolveCombat` 函數），新系統需要「等待玩家輸入 → 結算 → 推送結果 → 等待下一回合」的即時互動循環。多人組隊時，需要等所有人都下完指令才能結算。

**克服方案**：利用現有的 WebSocket 基礎設施（`wsServer.ts` 已有連線管理、認證、房間廣播），擴展出「戰鬥房間」概念。每場戰鬥是一個獨立的狀態機，戰鬥狀態存在伺服器記憶體中（不是資料庫），只在戰鬥結束時才寫入 DB 結算獎勵，保持流暢度。

### 挑戰二：多人同步一致性

**問題**：4 個玩家同時操作，網路延遲不同，如何確保所有人看到的戰鬥狀態一致？

**克服方案**：採用「伺服器權威」模式。所有戰鬥邏輯都在伺服器端計算，前端只負責「送出指令」和「顯示結果」。伺服器收齊所有指令（或超時）後統一結算，再廣播結果給所有玩家。

### 挑戰三：斷線重連

**克服方案**：戰鬥狀態存在伺服器記憶體中，玩家重連後可以透過 WebSocket 重新加入戰鬥房間。設定 30 秒重連視窗，超時後該玩家由 AI 託管。

### 挑戰四：前端戰鬥 UI

**克服方案**：分層設計（BattleScene → BattleField → ActionPanel → BattleLog → TeamStatus），用 CSS 動畫 + Framer Motion 做視覺效果，不需要 Canvas 或 WebGL。先做簡潔版本，動畫後加。

### 挑戰五：一隊獨佔機制

**問題**：如何確保同一時間只有一支隊伍在挑戰？

**克服方案**：在 DB 中記錄「當前挑戰隊伍 ID」和「開始時間」。進入前檢查是否有隊伍正在挑戰。設定最大戰鬥時間（例如 30 分鐘），超時自動判定失敗並釋放。

## 九、風險評估

| 風險 | 影響 | 應對方案 |
|------|------|----------|
| 回合制引擎開發時間超預期 | 延後所有後續功能 | 先做單人版，組隊後加 |
| WebSocket 多人同步 Bug | 戰鬥體驗差 | 優先保證單人流暢，多人逐步優化 |
| 前端戰鬥 UI 過於複雜 | 開發時間膨脹 | 先用簡潔文字+按鈕版本，動畫後加 |
| 數值平衡困難 | Boss 太難或太簡單 | 利用現有數值平衡儀表板，開放後台調整 |
| 伺服器負載增加 | 多人同時戰鬥卡頓 | 戰鬥狀態存記憶體，減少 DB 操作 |

## 十、已確認的設計決策

以下為老闆已確認的設計方向：

1. **戰鬥模式**：魔力寶貝式，每回合 30 秒，所有隊員同時下指令後統一結算。超時自動判定（HP 低防禦，否則攻擊/技能）
2. **惡魔城挑戰次數**：不限次數，未攻陷前可無限進攻
3. **世界 Boss 出現方式**：固定在地圖節點上，常駐存在
4. **惡魔城難度**：五座城池有不同風格難度（AI 生成）
5. **撤退機制**：撤退從第一關重新開始
6. **失敗處罰**：惡魔城 = 惡魔城詛咒（AI 發想負面狀態），世界 Boss = 掉經驗值
7. **獨佔機制**：全服一次只能有一隊進行挑戰，先打完就結束，需等冷卻時間
