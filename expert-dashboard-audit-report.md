> 本報告基於 8 個前端頁面、1 個共用佈局元件、1 個後端路由器（1,846 行）的完整程式碼掃描，涵蓋功能盤點、UI/UX 品質評估、與 /admin 後台的差距比較、以及分優先級的改善方向建議。

## 二、各頁面詳細分析

### 2.1 ExpertLayout（共用佈局元件，218 行）

ExpertLayout 是專家後台的統一佈局框架，負責路由守衛和響應式導航。它的設計品質在整個系統中算是最高的一個元件。

**已實現功能：**
- 角色守衛：僅 `expert` 或 `admin` 角色可進入，否則自動導回首頁
- 桌機版：左側固定側欄（w-56），含 Logo、5 項導航、底部「返回前台」和「管理後台」（僅 admin 顯示）
- 手機版：頂部標題列（含快捷操作按鈕插槽 `headerAction`）+ 底部五分頁導覽列
- 訂單紅點徽章：即時查詢待處理訂單數量，30 秒 staleTime + 60 秒 refetchInterval
- iOS 安全區域適配：底部導覽列加入 `env(safe-area-inset-bottom)`
- 活躍指示條：底部導覽列的活躍項目有頂部色條 + 圖示光暈效果

**品質評估：**
ExpertLayout 的整體設計水準相當不錯，響應式切換流暢，導航結構清晰。與 AdminLayout 相比，ExpertLayout 在手機版的設計上更為完善（AdminLayout 沒有底部導覽列），但在桌機版的側欄功能上較為簡潔。

**存在問題：**

| 問題 | 嚴重度 | 說明 |
|------|--------|------|
| 載入狀態使用 spinner 而非骨架屏 | 低 | 身份驗證中顯示旋轉圈，與 admin 後台的骨架屏風格不一致 |
| 缺少用戶資訊顯示 | 低 | 側欄底部沒有顯示當前登入用戶的名稱或頭像 |
| 導航項目固定 5 項 | 低 | 未來擴展（如收入統計、評價管理）需修改 NAV_ITEMS 陣列 |

### 2.2 儀表盤（ExpertDashboard.tsx，247 行）

儀表盤是專家登入後的第一個畫面，提供快速概覽和導航入口。

**已實現功能：**
- 初始化引導：未建立個人品牌時，顯示三步驟引導卡片（個人品牌 → 服務項目 → 行事曆）
- 統計卡片：服務項目數、待確認訂單數、已確認訂單數、平均評分
- 快速入口：三張卡片連結到個人品牌、服務項目、行事曆
- 金流串接預告：「即將開放」提示區塊，說明目前付款方式為手動上傳憑證
- 最近訂單：顯示最近 5 筆訂單的摘要

**品質評估：**
儀表盤的資訊架構合理，初始化引導設計得體。但與 admin 儀表盤（396 行，含圖表和多維度統計）相比，專家儀表盤的數據深度明顯不足。

**存在問題：**

| 問題 | 嚴重度 | 說明 |
|------|--------|------|
| 缺少收入統計 | 中 | 沒有本月/歷史收入數據，專家無法掌握營收狀況 |
| 缺少評價摘要 | 中 | 只顯示平均評分，沒有最近評價列表或評價趨勢 |
| 統計卡片無載入狀態 | 低 | 數據載入時統計卡片直接顯示 0 或空值，沒有骨架屏 |
| statusColor/statusLabel 定義在元件內部 | 低 | 與其他頁面重複定義，應抽取為共用常數 |
| 最近訂單無分頁 | 低 | 固定顯示前 5 筆，無法查看更多 |

### 2.3 個人品牌編輯（ExpertProfile.tsx，533 行）

個人品牌頁面是專家系統中功能最豐富的編輯器之一，支援多種內容格式。

**已實現功能：**
- 照片管理：頭像上傳（圓形裁切預覽）+ 封面圖片上傳（橫幅預覽），支援 hover 覆蓋上傳按鈕
- 基本資料：公開名稱、頭銜/職稱、專屬網址（slug，含格式驗證）
- 個人介紹：三種模式切換（純文字 / HTML 編輯 / 即時預覽），HTML 模式有安全過濾（SafeHtml 元件）
- 專業領域：32 個預設標籤 + 自訂標籤輸入，支援 Enter 快速新增
- 服務設定：最低/最高收費、服務語言、諮詢方式（視訊/語音/文字/面對面）
- 手機版快捷儲存：headerAction 插槽放置儲存按鈕

**品質評估：**
這是專家後台中設計最精細的頁面，HTML 介紹編輯器和安全過濾機制都考慮得很周到。不過 SafeHtml 的過濾邏輯使用正則表達式，在面對複雜的 XSS 攻擊向量時可能不夠穩固。

**存在問題：**

| 問題 | 嚴重度 | 說明 |
|------|--------|------|
| SafeHtml XSS 防護不完整 | 高 | 正則過濾可被繞過（如 `<img onerror=...>`），建議改用 DOMPurify |
| slug 唯一性檢查未實作 | 中 | 前端有 `slugStatus` 狀態但沒有實際呼叫後端驗證 API |
| 表單無驗證回饋 | 中 | publicName 為必填但沒有紅框/錯誤提示，只靠後端 zod 驗證 |
| 圖片上傳無進度指示 | 低 | 大圖片上傳時用戶看不到進度，只有按鈕 disabled |
| 缺少「預覽公開頁面」按鈕 | 低 | 專家無法在編輯後直接預覽自己的公開頁面效果 |

### 2.4 服務項目管理（ExpertServices.tsx，476 行）

服務項目管理是專家設定諮詢服務的核心頁面，設計品質在整個系統中屬於上乘。

**已實現功能：**
- 服務列表：分「上架中」和「已停用」兩組顯示，含統計摘要（總數/上架數/最低起價）
- CRUD 操作：新增/編輯（Dialog 表單）、刪除（確認 Dialog）、啟用/停用（一鍵切換）
- 表單設計：時長預設值（30/45/60/90/120 分鐘）+ 自訂輸入、價格預設值 + 自訂輸入
- 服務卡片：根據類型和價格顯示不同圖示，hover 時顯示操作按鈕
- 空狀態：精心設計的空狀態引導

**品質評估：**
這是專家後台中 UI 完成度最高的頁面。卡片設計精美，操作流程直覺，預設值選擇器降低了輸入門檻。唯一的遺憾是缺少服務排序功能。

**存在問題：**

| 問題 | 嚴重度 | 說明 |
|------|--------|------|
| 缺少服務排序 | 低 | 無法拖拽或手動調整服務在前台的顯示順序 |
| 刪除確認使用 Dialog 但非 ConfirmDialog | 低 | 與 admin 後台的 ConfirmDialog 元件風格不統一 |
| 服務說明不支援富文本 | 低 | 只有純文字 Textarea，無法加粗或分段 |
| 載入骨架屏過於簡陋 | 低 | 只有兩個灰色方塊 animate-pulse，沒有結構化骨架 |

### 2.5 行事曆管理（ExpertCalendar.tsx，762 行）

行事曆是專家後台中最複雜的頁面，整合了時段管理、訂單查看和活動公告三大功能。

**已實現功能：**
- 月曆視圖：自製月曆格子，每日顯示可預約/已預約/活動/訂單的彩色圓點指示
- 時段管理：單日新增 + 批次新增（多日同時段），支援 30 分鐘間隔的時間選擇器
- 活動公告：支援線下活動/線上活動/公告三種類型，含地點、人數上限、費用設定
- 日期詳情：點選日期後顯示該日的時段列表、訂單列表、活動列表
- 預約邏輯說明：藍色提示框解釋「先付款先佔位」的機制
- 月曆圖例：底部圖例說明各色圓點的含義
- 本月訂單總覽：底部卡片列出本月所有訂單

**品質評估：**
行事曆的功能覆蓋面很廣，月曆視圖的資訊密度設計得當。但 762 行的程式碼量暗示這個頁面承擔了過多責任，未來維護成本較高。

**存在問題：**

| 問題 | 嚴重度 | 說明 |
|------|--------|------|
| 頁面過於龐大（762 行） | 中 | 月曆、時段管理、活動管理應拆分為子元件 |
| 缺少「每週重複」時段功能 | 中 | 批次新增只能選多個日期，無法設定「每週一到五 9:00-17:00」的重複規則 |
| 月曆在手機上格子過小 | 中 | 7 欄格子在小螢幕上很擠，圓點指示幾乎看不清 |
| 活動刪除無確認 | 低 | 點擊垃圾桶圖示直接刪除，沒有確認步驟 |
| 時段刪除無確認 | 低 | 同上，直接刪除 |
| 缺少活動編輯功能 | 低 | 只能新增和刪除，無法編輯已建立的活動 |

### 2.6 訂單管理（ExpertOrders.tsx，378 行）

訂單管理是專家處理預約訂單的核心工作台。

**已實現功能：**
- 狀態篩選：全部/待確認/已確認/已完成/已取消，含待確認數量徽章
- 訂單卡片：展開/收合設計，顯示服務名稱、用戶名、日期時間、金額
- 訂單詳情：展開後顯示訂單編號、用戶備註、付款憑證預覽
- 操作按鈕：確認預約（含附加訊息）、標記完成、與用戶溝通、取消訂單
- 付款憑證預覽：全螢幕圖片覽器
- 確認/取消 Dialog：含附加訊息/取消原因輸入

**品質評估：**
訂單管理的操作流程設計合理，展開/收合的互動模式節省了畫面空間。但與 admin 後台的訂單管理（adminListAllBookings）相比，缺少搜尋和排序功能。

**存在問題：**

| 問題 | 嚴重度 | 說明 |
|------|--------|------|
| 缺少搜尋功能 | 中 | 無法按用戶名或服務名搜尋訂單 |
| 缺少排序功能 | 中 | 無法按日期、金額排序 |
| 缺少分頁 | 中 | 訂單量大時一次載入全部，效能堪憂 |
| 載入狀態使用 Loader2 spinner | 低 | 與 admin 後台的骨架屏風格不一致 |
| 「標記完成」無確認步驟 | 低 | 直接呼叫 mutation，沒有確認 Dialog |
| 付款憑證預覽是自製 overlay | 低 | 沒有使用 Dialog 元件，點擊外部關閉的 UX 不夠直覺 |

### 2.7 我的預約（MyBookings.tsx，495 行）

我的預約是用戶端的訂單管理頁面，讓消費者查看和管理自己的預約。

**已實現功能：**
- 狀態篩選：全部/待付款/已確認/已完成/已取消
- 訂單卡片：頂部色條、狀態徽章、金額、時間、備註
- 付款流程：付款說明 Dialog（ATM 轉帳資訊 + 付款方式選擇 + 上傳憑證）
- 評價系統：五星評分 + 文字評論
- 取消訂單：含取消原因輸入
- 聯絡老師：跳轉到訊息聊天室

**品質評估：**
用戶端的訂單管理功能完整，付款流程的引導設計得體。但 ATM 轉帳資訊目前是「待設定」的佔位符，需要後台管理介面來配置。

**存在問題：**

| 問題 | 嚴重度 | 說明 |
|------|--------|------|
| ATM 轉帳資訊為硬編碼佔位符 | 高 | 銀行/帳號/戶名都是「待設定」，應從 systemSettings 讀取 |
| 付款憑證上傳使用 base64 | 中 | 大圖片轉 base64 會佔用大量記憶體，應改用 FormData 直傳 |
| 缺少訂單詳情頁 | 低 | 所有資訊擠在卡片中，沒有獨立的訂單詳情頁 |
| 15 天保留說明只在付款 Dialog | 低 | 應在更顯眼的位置提醒用戶 |

### 2.8 訂單訊息聊天室（Messages.tsx，298 行）

訊息聊天室是用戶與專家在訂單成立後的溝通頻道。

**已實現功能：**
- 訂單綁定：透過 URL 參數 `bookingId` 綁定特定訂單
- 訊息氣泡：區分自己/對方的訊息，含頭像、發送者名稱、時間戳
- 已讀狀態：自己發送的訊息顯示「已讀」或「✓」
- 自動捲動：新訊息自動捲動到底部
- 輪詢更新：每 5 秒 refetch 一次
- 取消訂單鎖定：已取消的訂單無法發送訊息
- 資訊提示：訂單專屬頻道說明 + 15 天保留說明

**品質評估：**
聊天室的基本功能可用，但與現代即時通訊的體驗相比有明顯差距。5 秒輪詢的延遲感較重，且缺少打字中指示、圖片發送等常見功能。

**存在問題：**

| 問題 | 嚴重度 | 說明 |
|------|--------|------|
| 使用輪詢而非 WebSocket | 中 | 5 秒延遲影響即時性，且浪費伺服器資源 |
| 缺少圖片/檔案發送 | 中 | 命理師可能需要發送命盤圖片給用戶 |
| 缺少打字中指示 | 低 | 無法知道對方是否正在輸入 |
| 訂單資訊查詢效率低 | 低 | 同時查詢 myBookings 和 listMyBookings 兩個 API 來找訂單資訊 |
| 缺少訊息搜尋 | 低 | 無法搜尋歷史訊息 |
| `(msg as any).isRead` 型別不安全 | 低 | 使用 `as any` 繞過型別檢查 |

**品質評估：**
後端路由器的功能覆蓋面很完整，56 個 procedure 涵蓋了專家平台的所有核心業務。但 1,846 行的單檔案已經超過了建議的 150 行上限，應該拆分為多個子路由器。

**存在問題：**

| 問題 | 嚴重度 | 說明 |
|------|--------|------|
| 單檔案過大（1,846 行） | 高 | 遠超 150 行建議上限，應拆分為 expertProfile, expertService, expertBooking, expertMessage 等子路由器 |
| confirmPayment 與 expertConfirmBooking 功能重疊 | 中 | 兩個 procedure 都用於確認訂單，語義不清 |
| 缺少收入統計 API | 中 | 沒有提供專家收入彙總的 procedure |
| 缺少評價回覆 API | 低 | 專家無法回覆用戶的評價 |
| 缺少服務排序 API | 低 | 沒有調整服務顯示順序的 procedure |
| 測試覆蓋率極低 | 中 | 只有 1 個測試檔案（63 行），僅測試 adminUpdateExpertProfile 的輸入構建 |

## 三、與 /admin 後台的差距比較

以下是專家後台與 admin 後台在 UI/UX 標準化方面的差距分析：

| 比較項目 | /admin 後台（v5.23） | /expert 後台（現狀） | 差距 |
|----------|---------------------|---------------------|------|
| **統一佈局** | AdminLayout 套用所有 15 頁 | ExpertLayout 套用所有 5 頁 | ✅ 已達標 |
| **確認對話框** | ConfirmDialog 元件替換 18 處 confirm() | 使用 Dialog 元件（非 ConfirmDialog） | ⚠️ 風格不統一 |

## 五、缺失功能盤點

以下是目前專家系統中尚未實現但對業務運營至關重要的功能：

## 八、總結

天命聯盟的專家後台系統在功能覆蓋面上已經相當完整，56 個 API procedure 涵蓋了從專家入駐、服務管理、時段設定、訂單處理到訊息溝通的完整業務流程。ExpertLayout 的響應式設計（尤其是手機版的底部導覽列）甚至優於 AdminLayout。

但在 UI/UX 標準化方面，專家後台與 admin 後台（v5.23）之間存在明顯差距：缺少骨架屏載入、缺少統一的確認對話框、狀態常數重複定義 5 處、測試覆蓋率極低（1 個測試 vs admin 的 25 個）。此外，SafeHtml 的 XSS 防護和 ATM 轉帳資訊的硬編碼是兩個需要優先處理的安全/功能問題。

建議優先執行 P0 和 P1 項目，將專家後台的 UI/UX 品質對齊到 admin 後台的水準，然後再逐步補齊收入統計、評價管理等業務功能。
